# 开口有招 · 系统模块架构设计文档

> 队伍：忍不了小队 | 赛道：生存时间
> 选题标题：开口有招：真实案例驱动的女性沟通策略库与表达训练场
> 一句话简介：面对职场刁难、社交冒犯和群聊施压，女性常因当下语塞而错失回应；用真实案例拆解、应对策略和模拟练习，帮助她们清晰、体面且坚定地表达自己。

> **⚠️ 风险修订说明**：本版本针对三项风险做了调整——① 法律支持库改为纯静态、人工预审内容，AI 不现场生成任何法律表述；② 真实案例库改为基于常见真实沟通模式提炼的"复合案例"，不对应任何具体真实帖子或博主；③ 新增"关系与权力维度"判断，避免对上级/长辈等场景无差别推荐同一种语气。具体改动见对应模块。

---

## 一、整体架构说明

系统分为三层，下层为上层提供支撑，数据自下而上流动：

```
┌─────────────────────────────────────────────┐
│  第三层：用户功能层（用户直接使用的入口）        │
│  话术生成助手 ｜ 社区广场 ｜ 情景练习            │
└─────────────────────────────────────────────┘
                    ▲
┌─────────────────────────────────────────────┐
│  第二层：AI 能力层（处理引擎）                  │
│  场景分类器 ｜ 内容生成引擎 ｜ 语音交互引擎        │
└─────────────────────────────────────────────┘
                    ▲
┌─────────────────────────────────────────────┐
│  第一层：知识与数据层（内容基础）                │
│  应对框架库 ｜ 真实案例库 ｜ 法律支持库           │
└─────────────────────────────────────────────┘
```

**关键闭环**：社区广场里被认可的优质案例，会沉淀回真实案例库，案例库越来越大 → 生成引擎能参考的素材越来越好 → 生成结果质量越来越高 → 吸引更多人来用/来发案例。这是整个产品能"越用越聪明"的核心飞轮。

## 二、模块总览表

| 层级 | 模块 | 一句话定位 |
|---|---|---|
| 知识与数据层 | ① 应对框架库 | 沉淀"遇到 XX 类型冲突该怎么应对"的通用方法论 |
| 知识与数据层 | ② 真实案例库 | 收录基于真实沟通模式提炼的复合案例，具体、可复用（天然匿名，无可追溯来源） |
| 知识与数据层 | ③ 法律支持库 | 给部分回应提供法律依据，增加说服力 |
| AI 能力层 | ④ 场景分类器 | 判断用户情境属于什么冲突类型、什么场景，以及双方的关系/权力对比 |
| AI 能力层 | ⑤ 内容生成引擎 | 核心差异化能力，结合①②③生成个性化回应 |
| AI 能力层 | ⑥ 语音交互引擎 | 为情景练习提供语音输入输出能力 |
| 用户功能层 | ⑦ 话术生成助手 | 用户输入情境，获得可用的应对话术 |
| 用户功能层 | ⑧ 社区广场 | 浏览/发布/讨论情境和案例，UGC 内容来源 |
| 用户功能层 | ⑨ 情景练习 | 模拟真实场景，练习临场应对能力 |

## 三、知识与数据层详解

### ① 应对框架库

**定位**：这是你自己沉淀的方法论，不需要 AI 生成，是产品的"骨架"，决定了生成结果是否有章法。

**设计逻辑**：按冲突类型分类，每类配一套通用应对逻辑（公式化的话术结构），而不是具体的话。

**数据结构**：
```json
{
  "id": "F001",
  "conflict_type": "职场故意刁难",
  "signal_words": ["阴阳怪气", "当众否定", "模糊要求后甩锅"],
  "strategy_name": "命名行为—设立边界—转移主动权",
  "formula": "我注意到你在{具体行为}，这让我觉得{感受/影响}，接下来我希望{明确要求}",
  "tone_variants": ["温和坚定", "犀利反击", "职场体面"]
}
```

**建议覆盖的冲突类型**（先做这 5-6 类，覆盖大部分场景）：

| 冲突类型 | 核心应对逻辑 |
|---|---|
| 职场故意刁难/PUA | 不接情绪，要求对方具体化，把举证责任推回去 |
| 道德绑架 | 承认对方的感受，但拒绝为其感受负责 |
| 当众人身攻击 | 先设界限（"这样说话我不接受"），再决定是否继续对话 |
| 造谣/构陷 | 只回应事实，不卷入情绪对骂，必要时提及后果 |
| 阴阳怪气/冷暴力 | 直接把潜台词说破，逼对方明说 |
| 家庭催促/道德捆绑（催婚催生等） | 表达理解+清晰划界+重复法（不被带偏节奏） |

**效果**：生成引擎每次生成回应前，先根据分类结果匹配到对应的框架条目，保证输出"有逻辑支撑"而不是即兴发挥。

---

### ② 真实案例库

**定位**：产品最核心的差异化资产。别人是"通用怼人语录"，你是"扎根真实沟通模式、可复用、还在生长"的案例库。

**⚠️ 风险修订**：种子案例**不采用"抓取某条具体真实帖子后改写"的做法**——即便做了匿名化，删掉名字也不等于解决了对原作者内容的版权/隐私边界问题。改为：**基于大量公开、真实存在的沟通冲突模式，提炼撰写成"复合案例"**，每条案例是团队总结常见情境后原创写出的典型场景，不对应任何一条具体的真实帖子或博主。对外统一说法是"我们研究了大量真实沟通场景后提炼的典型案例"，不说"这是从 XX 博主那来的"——这样处理反而比"匿名化"更彻底，因为根本不存在可追溯的原始来源。

**数据结构**：
```json
{
  "id": "C0231",
  "scenario_type": "职场",
  "context": "会议上被同事当众否定方案",
  "original_line": "这个方案是不是有点想当然了，能力不行就别硬撑",
  "response": "具体是哪部分让你觉得想当然？我很乐意听你的建议，如果只是感受，那我们先看数据。",
  "why_effective": "不接情绪，把焦点转向要求对方给出具体依据，主动权和举证责任被推回对方",
  "technique_tags": ["转移焦点", "要求具体化", "职场体面"],
  "is_composite": true,
  "source_type": "官方复合案例 / 用户上传 / 广场沉淀",
  "likes": 128,
  "status": "已收录"
}
```
（新增 `is_composite` 字段：标记这条案例是团队基于常见模式原创提炼的"复合案例"，还是用户上传的真实亲历经历。两者在广场和案例库里应有视觉区分，避免混淆"虚构典型案例"和"某个真实用户的真实经历"。）

**案例来源与运营机制**（三个来源叠加）：
1. **官方种子**：团队基于对大量真实、公开沟通冲突模式的观察，原创撰写 10-15 条复合案例，作为 MVP 冷启动数据，标记 `is_composite: true`
2. **用户主动上传**：用户可以上传自己真实亲历的经历，AI 自动帮忙拆解结构化（提取情境类型、技巧标签、为什么有效），标记 `is_composite: false`，上传时需用户确认已自行做过匿名化处理，进阶版再开放
3. **广场沉淀**：社区广场里点赞/认可度高的回复，经简单审核后正式收录进案例库（详见模块⑧）

**效果**：生成引擎会根据用户当前情境，检索出 2-3 条最相关的案例作为参考（few-shot），让生成结果"贴近真实沟通场景"而不是凭空编。

### ③ 法律支持库

**定位**：给部分场景的回应加上法律依据，让建议从"情商话术"升级为"你有权这样做"。

**⚠️ 风险修订**：这是产品里风险最高的一块——LLM 对法条编号、具体适用条件这类细节很容易"一本正经地编"，一旦编错，恰恰会摧毁产品最想建立的"专业可信"人设。因此调整为：**法律内容 100% 由人工预先审核撰写、静态存储，AI 不参与生成或改写任何法律相关表述**。AI 唯一的角色是判断用户情境是否命中某条已审核内容的 `applicable_tags`，命中则原样插入这条预先写好的静态文本，不命中就不展示，绝不临时编造新的法律解读。

**触发边界原则**：不是所有情境都要往法律上靠。大部分情境（比如家庭催婚、朋友间开过分的玩笑、普通的阴阳怪气）本质是**道德和情感范畴**，只需要应对框架库和案例库提供话术支持，**不强行关联法律内容**——硬凑一个法条反而显得生硬、甚至无谓地把关系冲突升级成法律对抗。只有情境**真正触及法律边界**时，才展示法律参考卡片。判断依据就是 `applicable_tags` 是否命中，命中即出现，不命中则该情境下完全不出现法律相关内容，不用任何"泛法律化"的兜底话术。

参考对照：

| 情境类型 | 是否触及法律 | 处理方式 |
|---|---|---|
| 家庭催婚催生、道德绑架 | 否，纯粹家庭/道德范畴 | 只给应对话术，不展示法律参考 |
| 社交场合被起哄、开过分玩笑 | 通常否 | 只给应对话术 |
| 职场恶意调岗、拖欠加班费、违法加班 | 是，涉及劳动权益 | 展示法律参考卡片 |
| 被当众造谣、诽谤 | 是，涉及名誉权 | 展示法律参考卡片 |
| 购物被诱导消费、强制搭售 | 是，涉及消费者权益 | 展示法律参考卡片 |
| 公共场合被言语骚扰、威胁 | 是，可能涉及治安管理处罚 | 展示法律参考卡片 |

**数据结构**：
```json
{
  "id": "L012",
  "law_name": "示例：劳动法相关条款",
  "plain_summary": "用大白话解释这条法律在说什么，普通人能懂（人工撰写并审核，不由 AI 生成）",
  "applicable_tags": ["职场PUA", "违法加班", "恶意调岗"],
  "usage_note": "仅作为口头沟通时的底气参考，不构成正式法律意见，具体情况建议咨询专业律师",
  "reviewed_by": "团队人工审核（建议找有法律背景的朋友复核一遍）"
}
```

**建议覆盖的法律领域**（先做 3-4 类最高频的）：
- 职场骚扰/恶意刁难相关（劳动权益方向）
- 名誉权/诽谤相关（被造谣、被当众辱骂）
- 消费者权益相关（购物消费场景被为难）
- 治安管理处罚相关（公共场合被言语攻击/骚扰）

**重要设计原则**：
- 每条法律引用旁边必须标注"仅供参考，不构成正式法律意见，具体情况建议咨询专业律师"
- 展示形式是**静态卡片**，不是聊天式的实时生成内容，视觉上要和 AI 生成的话术明显区分（比如单独的卡片样式+图标），让用户清楚知道"这部分是人工审核过的固定内容，不是 AI 现场编的"
- MVP 阶段只做 2-3 条团队非常有把握、简单直接的条目，宁可少而准，不做不确定的内容

**效果**：当用户情境真正命中相关法律标签时，系统会附上一条预先审核好的法律参考卡片，增加话术的底气和可信度；对纯道德/情感类情境则完全不出现法律相关内容，避免不必要的"泛法律化"。

## 四、AI 能力层详解

### ④ 场景分类器

**定位**：所有请求的第一道处理，决定后续调用哪些框架和案例。

**分类维度**：
- 冲突类型（对应框架库的分类）
- 场景类型（职场 / 家庭 / 社交餐饮 / 购物消费 / 陌生人冲突）
- 紧急程度（是否需要立刻回应 vs 提前准备）
- **⚠️ 新增：关系与权力维度**（对方是平级/陌生人，还是上级/长辈等权力不对等的一方）——这个维度之前完全没设计，但很关键：同一句话，对平级同事和对直属领导，能用的"犀利反击版"安全程度完全不同，产品不能无差别推荐

**实现方式（MVP 简化版）**：直接用一次 LLM 调用做分类，要求模型输出结构化 JSON，不需要额外训练分类模型：
```
系统提示：请判断以下情境属于哪种冲突类型、场景类型，以及双方的关系/权力对比，只输出 JSON：
{"conflict_type": "...", "scene_type": "...", "urgency": "high/low", "power_dynamic": "平级/上级或长辈/陌生人/亲密关系"}
用户情境：{用户输入}
```

**效果**：分类结果直接决定去框架库和案例库里检索哪些条目，其中 `power_dynamic` 会传给内容生成引擎，用于给每个语气版本加上适用性提示（见模块⑤）。

---

### ⑤ 内容生成引擎（核心能力）

**定位**：整个产品的核心差异化能力，决定生成质量的好坏。

**Prompt 设计逻辑**：
```
系统角色：你是一个应对教练，帮用户在被冒犯/刁难/为难时给出体面又有力的回应。

已知信息：
- 匹配到的应对框架：{框架库条目}
- 相关案例（仅供参考语气和结构，不要照抄）：{2-3条案例库复合案例}
- 分类结果中的关系与权力维度：{power_dynamic}

用户情境：{用户输入}

请生成 3 个不同语气版本的回应（温和坚定版 / 犀利反击版 / 职场体面版），
每个版本附一句话解析"为什么这样回应有效"，
并根据 {power_dynamic} 给出一句"适用提示"（比如对方是上级/长辈时，提示犀利反击版可能有额外风险，建议优先考虑体面版）。
只输出 JSON 数组。

⚠️ 重要限制：不要生成任何具体法律条文、法条编号或法律解读。如果情境命中了法律支持库里预先审核好的静态条目，由系统在这一步之外单独插入固定文本，你（模型）不参与法律内容的生成。
```

**输出效果示例**：

> 用户输入："我妈总说'你都多大了还不结婚，再挑就没人要了'"（关系：长辈）
>
> - **温和坚定版**："妈，我知道你是担心我，但结婚这件事我想按自己的节奏来，你越催我反而压力更大。" —— *先接住对方情绪，再温和划界，避免正面冲突*　适用提示：面对长辈，这个版本通常最稳妥
> - **幽默化解版**："妈你这话我都能倒背如流了，要不咱俩打个赌，我明年这时候还没对象请你吃火锅？" —— *用玩笑转移紧张气氛，同时也传达了"这个话题我听腻了"*
> - **直接设界版**："这个话题我们已经聊过很多次了，我希望你尊重我的决定，不然我会有点想挂电话。" —— *明确说出边界和后果*　适用提示：语气较强，建议在多次沟通无效、关系能承受一定摩擦时再使用

**效果**：这是用户使用完直接能截图/复制走用的核心产出；加入适用提示后，产品从"无差别给建议"变成"帮用户判断哪个建议在当前关系里更安全"，这本身也是很好的差异化和责任感体现。

---

### ⑥ 语音交互引擎

**定位**：为情景练习模块服务，让练习更接近真实反应速度。

**组成**：
- ASR（语音转文字）：用户说出自己的应对，转成文字给生成引擎评价
- TTS（文字转语音）：AI 扮演的"对方"用语音说话，营造真实压迫感

**技术选型建议**：优先用浏览器自带的 Web Speech API（免费、无需额外接入），能跑通概念即可；如果时间允许，再接入更自然的第三方语音合成服务提升音色真实感。

**进阶功能**：识别用户语音里的情绪/语速变化，练习结束后给出"你刚才听起来有点慌，语速快了"这类反馈。

## 五、用户功能层详解

### ⑦ 话术生成助手（核心页面）

**用户流程**：
1. 打开应用，进入"求助"入口
2. 输入具体情境（文字，或未来支持截图上传）
3. 系统自动分类（用户无感知，后台完成）
4. 展示 3 个语气版本的回应，可切换 tab 查看，每个版本下方附一句"适用提示"（比如是否更适合对上级/长辈使用）
5. 每个回应下方可展开"为什么有效"的解析
6. 一键复制 / 收藏 / 分享到广场

**页面效果**：类似"输入框 + 结果卡片"的简洁交互，结果卡片用 tab 或左右滑动切换三种语气，避免信息堆砌。移动端优先，单手可操作。

---

### ⑧ 社区广场

**用户流程**：
1. 首页信息流：浏览别人发布的情境/提问，按场景类型或热度筛选
2. 点开某条情境，查看大家给出的回应、点赞数、评论
3. 可以给别人的情境提供自己的回应建议
4. 可以自己发布：**求助型**（我遇到这个情况，怎么回比较好）或 **分享型**（分享一个我看到/经历的好案例）
5. 高赞回应会被标记"已收录进案例库"，形成正向激励

**内容运营机制（简化版）**：
- 关键词过滤 + 基础人工/官方精选，避免低质或攻击性内容
- 点赞数达到阈值的回应，自动进入"待收录"队列，官方审核后正式入库

**页面效果**：类似小红书的信息流卡片 + 评论区结构，用户比较熟悉，学习成本低。

---

### ⑨ 情景练习

**基础版：选项分支剧情**
- AI 先生成一个具体开场情境（比如"你的领导突然在会议上说……"）
- 提供 2-4 个应对选项
- 选择后剧情按选择走向不同结果，并给出简短点评（"这个回应可能会让对方更强硬" / "这个回应体面地化解了冲突"）
- 支持"重新挑战"，尝试不同选项看不同结局

**进阶版：沉浸式互动**
- 不给预设选项，AI 实时扮演"对方"角色，用户自由输入/语音说出自己的应对
- AI 根据用户的回应动态调整"剧情走向"（对方态度可能软化或更强硬）
- 结束后给出总结评价：这次练习你的应对方式属于哪种风格、哪里可以更好

**场景类型覆盖**（建议按优先级逐步做）：

| 场景类型 | 示例练习情境 |
|---|---|
| 职场 | 会议上被当众否定、被恶意甩锅、被穿小鞋 |
| 家庭（对父母） | 催婚催生、被否定人生选择、被道德绑架 |
| 社交餐饮 | 饭局上被灌酒/开不合适的玩笑、被起哄 |
| 购物消费 | 被导购/客服态度恶劣、被诱导消费后维权被为难 |

**语音功能集成**：进阶版默认开启语音交互，用户可以直接说出应对，更贴近真实场景的临场反应速度。

**效果**：这个模块是把"知道该怎么说"变成"真的能脱口而出"的关键，是产品和"静态语录合集"最本质的区别。

## 六、模块间数据流转小结

1. 用户在**话术生成助手**或**情景练习**中输入情境
2. **场景分类器**判断冲突类型和场景类型
3. 系统据此从**应对框架库**取出对应策略、从**真实案例库**检索相关复合案例、从**法律支持库**匹配是否命中已审核的静态法律参考（不现场生成）
4. **内容生成引擎**（或**语音交互引擎**，用于练习模式）综合以上信息生成个性化输出
5. 用户获得结果后，可以选择分享到**社区广场**
6. 广场里的优质内容经审核后，反哺回**真实案例库**，完成闭环
